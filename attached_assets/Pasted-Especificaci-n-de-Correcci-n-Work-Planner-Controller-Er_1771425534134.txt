Especificación de Corrección: Work Planner Controller (Error 500)
Archivo: GestorOT.Controllers.LaborsController.cs (o donde se encuentre GetCalendarLabors)
Objetivo: Corregir la excepción Npgsql.ResolveTypeInfo asegurando que las fechas de filtro sean compatibles con el tipo de dato de PostgreSQL (timestamptz o date).
Instrucciones para el Agente:
1. Normalización de Fechas: Antes de entrar en la consulta LINQ, convierte los parámetros start y end a UTC explícito. Npgsql requiere esto para columnas timestamp with time zone.
2. Proyección de Datos (Select): El código en la imagen (línea 273) parece incompleto (corta en el Where). Debes agregar el .Select para transformar la entidad Labor al DTO LaborCalendarDto antes de materializar la lista.
3. Manejo de Nulos: Asegura que la proyección maneje posibles nulos en relaciones (l.Lot, l.WorkOrder) para evitar NullReferenceException.
Código Corregido (Reemplazar el método GetCalendarLabors):
[HttpGet("calendar")]
public async Task<ActionResult<List<LaborCalendarDto>>> GetCalendarLabors(
    [FromQuery] DateTime start, [FromQuery] DateTime end)
{
    // PASO 1: Corrección crítica para Npgsql
    // Npgsql falla si se envía DateTime.Kind=Unspecified a una columna timestamptz.
    // Forzamos UTC.
    var startUtc = DateTime.SpecifyKind(start, DateTimeKind.Utc);
    var endUtc = DateTime.SpecifyKind(end, DateTimeKind.Utc);

    var labores = await _context.Labors
        .AsNoTracking()
        .Include(l => l.Lot)
        .Include(l => l.WorkOrder)
        // Usamos las variables UTC corregidas
        .Where(l => (l.PlannedDate != null && l.PlannedDate >= startUtc && l.PlannedDate <= endUtc)
                 || (l.PlannedDate == null && l.ExecutionDate != null && l.ExecutionDate >= startUtc && l.ExecutionDate <= endUtc)
                 || (l.PlannedDate == null && l.ExecutionDate == null && l.CreatedAt >= startUtc && l.CreatedAt <= endUtc))
        // PASO 2: Proyección al DTO (Faltaba en la imagen o estaba implícito)
        .Select(l => new LaborCalendarDto
        {
            Id = l.Id,
            Title = l.Lot != null ? $"{l.Type} - {l.Lot.Name}" : l.Type, // Protección contra nulos
            Date = l.PlannedDate ?? l.ExecutionDate ?? l.CreatedAt,
            Status = l.Status,
            HasWorkOrder = l.WorkOrderId != null,
            // Asignar colores según estado/tipo
            ColorHex = l.WorkOrderId == null ? "#FFA500" : "#4CAF50" 
        })
        .ToListAsync();

    return Ok(labores);
}
Nota Adicional para el Agente:
Si la columna en base de datos PlannedDate está definida estrictamente como DATE (sin hora) y no TIMESTAMP, el filtro debe cambiar ligeramente para usar DateOnly.FromDateTime(start). Sin embargo, la corrección a UTC arriba es la solución estándar para el 90% de los errores de ResolveTypeInfo en este stack.
