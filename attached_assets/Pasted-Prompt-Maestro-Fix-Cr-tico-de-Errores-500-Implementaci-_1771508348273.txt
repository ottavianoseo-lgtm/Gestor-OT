Paso 2: Fix Raíz de los Errores 500 (Serialización AOT)
Los errores 500 se disparan al intentar enviar la respuesta JSON porque la compilación AOT eliminó los metadatos de los nuevos DTOs.
En AppJsonSerializerContext.cs (o tu contexto de Source Generator): Debes registrar explícitamente TODOS los modelos que devuelven los endpoints que están fallando.
[JsonSerializable(typeof(LaborDetailDto))]
[JsonSerializable(typeof(LaborDto))]
[JsonSerializable(typeof(List<LaborDto>))]
[JsonSerializable(typeof(List<LaborCalendarDto>))]
Si no haces esto, .NET 10 AOT siempre arrojará Error 500 al llamar a /api/stats/labors/{id} y /api/labors/unassigned.
Paso 3: Fix de NullReferenceException en Endpoints (Backend)
Debes reescribir los endpoints que fallan para evitar que EF Core explote al proyectar datos nulos (como lotes sin asignar).
En tu Controlador/Minimal API para Labores:

app.MapGet("/api/stats/labors/{id}", async (Guid id, AppDbContext db) =>
{

        .AsNoTracking()
        .Include(l => l.Lot) // Evita NullReferenceException

    // Mapeo seguro en memoria
    var dto = new LaborDetailDto

        PlannedDate = labor.PlannedDate,
        LotName = labor.Lot?.Name ?? "Labor Suelta (Sin Lote)",
        LotId = labor.LotId,
        ResponsibleName = "Sin Asignar", // Mapear a responsable si existe en tu DB
        Machinery = labor.Machinery ?? "N/A"
    };
    return Results.Ok(dto);
});

app.MapGet("/api/labors/unassigned", async (AppDbContext db) =>
{
    var unassignedLabors = await db.Labors
        .AsNoTracking()
        .Include(l => l.Lot)
        .Where(l => l.WorkOrderId == null)
        .Select(l => new LaborDto 
        {
            Id = l.Id,
            Type = l.Type,
            Status = l.Status,
            PlannedDate = l.PlannedDate,
            LotName = l.Lot != null ? l.Lot.Name : "Sin Lote" // Crítico
        })
        .ToListAsync();

    return Results.Ok(unassignedLabors);
});
