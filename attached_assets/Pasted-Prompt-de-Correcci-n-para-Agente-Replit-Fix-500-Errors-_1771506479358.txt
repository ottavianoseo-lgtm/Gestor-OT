Prompt de Corrección para Agente Replit: Fix 500 Errors en Labors y Stats
Contexto: El cliente Blazor está recibiendo errores 500 Internal Server Error al llamar a los endpoints GET /api/stats/labors/{id} y GET /api/labors/unassigned. Objetivo: Corregir las excepciones de proyecciones nulas en EF Core y asegurar la compatibilidad de serialización para Native AOT.
Aplica las siguientes tres correcciones en el backend:
1. Corrección de Serialización JSON (Native AOT)
Al usar compilación AOT en .NET 10, no se permite la reflexión. Los nuevos DTOs devueltos por estos endpoints deben estar registrados en el contexto del serializador.
• Acción: Ve a la clase AppJsonSerializerContext (o la clase que herede de JsonSerializerContext en tu proyecto compartido/cliente) y asegúrate de incluir explícitamente las colecciones y DTOs requeridos.
// Añadir estas líneas al JsonSerializerContext
[JsonSerializable(typeof(LaborDetailDto))]
[JsonSerializable(typeof(List<LaborDto>))] // O el DTO exacto que devuelva /unassigned
[JsonSerializable(typeof(LaborDto))]
2. Corrección Endpoint 1: /api/stats/labors/{id}
Este endpoint falla al intentar construir el detalle (posible NullReferenceException si el lote o el responsable son nulos).
• Acción: Revisa el controlador o Minimal API que maneja esta ruta y modifica la consulta LINQ para manejar propiedades de navegación nulas de forma segura.
app.MapGet("/api/stats/labors/{id}", async (Guid id, AppDbContext db) =>
{
    var labor = await db.Labors
        .AsNoTracking()
        .Include(l => l.Lot)
        // Asegurar que si hay tabla/entidad responsable se incluya, o omitir si es un simple string
        .Where(l => l.Id == id)
        .Select(l => new LaborDetailDto
        {
            LaborId = l.Id,
            Type = l.Type,
            Status = l.Status,
            PlannedDate = l.PlannedDate,
            // PROTECCIÓN CONTRA NULOS:
            LotName = l.Lot != null ? l.Lot.Name : "Sin Lote Asignado",
            LotId = l.LotId,
            // PROTECCIÓN CONTRA NULOS (Ajustar según cómo modelaste Responsible):
            ResponsibleName = l.Responsible != null ? l.Responsible.Name : "Sin Asignar",
            Machinery = l.Machinery ?? "N/A"
        })
        .FirstOrDefaultAsync();

    return labor is not null ? Results.Ok(labor) : Results.NotFound();
});
3. Corrección Endpoint 2: /api/labors/unassigned
Este endpoint busca labores sueltas (WorkOrderId == null). El error 500 suele ocurrir por la misma falta de validación de nulos al proyectar los lotes.
• Acción: Actualiza la consulta asegurando la carga de relaciones y manejo de nulos.
app.MapGet("/api/labors/unassigned", async (AppDbContext db) =>
{
    var unassignedLabors = await db.Labors
        .AsNoTracking()
        .Include(l => l.Lot)
        .Where(l => l.WorkOrderId == null) // Filtro clave
        .Select(l => new LaborDto 
        {
            Id = l.Id,
            Type = l.Type,
            Status = l.Status,
            PlannedDate = l.PlannedDate,
            // PROTECCIÓN CONTRA NULOS CRÍTICA:
            LotName = l.Lot != null ? l.Lot.Name : "Sin Lote" 
        })
        .ToListAsync();

    return Results.Ok(unassignedLabors);
});
